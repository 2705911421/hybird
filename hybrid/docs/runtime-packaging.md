# Runtime packaging

The supported Phase 9 Runtime distribution is a PyInstaller standalone executable built separately on Windows, macOS, and Linux. It embeds CPython and Python dependencies; end users do not create a venv. Cross-compiling is not supported.

`hybrid/story-runtime/story-runtime.spec` is the reproducible build entry. Release CI installs the version-constrained release inputs, builds `story-runtime` (`story-runtime.exe` on Windows), runs `--help`, uploads an OS-tagged artifact, emits a SHA-256 checksum, and attaches the source, SBOM and license material. Dependency resolution is not fully locked yet and remains part of the release reproducibility review.

The Windows development-workspace build was exercised with both `--help` and a real loopback `/health` request. This proves the standalone starts on that machine; it is not clean-machine or cross-platform evidence.

InkOS remains the product shell and supports Node 22. Runtime tokens are generated by the trusted Node process and passed only in the child environment. The Studio browser receives only server-side proxy responses. Runtime binds loopback, uses a discovered port, performs version/schema handshake, has one PID owner, removes stale PID files, restarts with capped exponential backoff, and shuts down its process group.

Those lifecycle behaviors are implemented in the exported core process manager, but Studio and CLI bootstrap do not instantiate it yet. Until that integration selects the OS-specific bundled executable and injects the discovered URL server-side, users do not have an accepted one-click combined App/Runtime launch path.

The current repository does not yet contain a native GUI installer that bundles Node and registers uninstall metadata. The release workflow produces portable OS packages and a standalone Runtime, but a clean-machine signed installer/notarized package remains a release blocker; portable archives must not be described as an installer.
